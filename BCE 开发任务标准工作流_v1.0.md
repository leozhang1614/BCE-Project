## 🎯 核心流程

```
┌─────────┐     ┌─────────┐     ┌─────────┐
│  开发   │ ──→ │  验收   │ ──→ │  审核   │
│ (开发者)│     │(负责人) │     │ (执矩)  │
└─────────┘     └─────────┘     └────┬────┘
                                     │
                            ┌────────┴────────┐
                            │                 │
                     ┌──────▼──────┐   ┌────▼────┐
                     │ 审核通过    │   │审核不通过│
                     │ 任务完成    │   │         │
                     └─────────────┘   └────┬────┘
                                            │
                                            │ 回退到开发节点
                                            │ (重点！)
                                            ▼
                                     ┌─────────────┐
                                     │  重新开发   │
                                     │  重新验收   │
                                     │  重新审核   │
                                     └─────────────┘
```

## 📋 工作流节点

### 1. 开发节点

**角色：** 开发者

**职责：**
- ✅ 功能实现
- ✅ 单元测试
- ✅ 自测通过
- ✅ 编写开发文档

**输出：**
- 源代码
- 自测报告
- 开发文档

**完成标准：**
- 代码功能完整
- 自测通过
- 文档齐全

---

### 2. 验收节点

**角色：** 项目负责人

**职责：**
- ✅ 功能验证
- ✅ 需求符合度检查
- ✅ 用户体验测试
- ✅ 编写验收报告

**输出：**
- 验收报告
- 问题清单（如有）

**完成标准：**
- 功能符合需求
- 用户体验良好
- 无严重 Bug

---

### 3. 审核节点

**角色：** 执矩（CLO）

**职责：**
- ✅ 代码合规性审核
- ✅ 系统稳定性评估
- ✅ OpenClaw 功能影响评估
- ✅ 安全风险评估

**审核标准：**
1. **代码合规**
   - 符合代码规范
   - 无硬编码配置
   - 有完整的错误处理

2. **系统稳定**
   - 不影响系统稳定性
   - 无内存泄漏
   - 无性能问题

3. **OpenClaw 兼容**
   - 不影响 OpenClaw 核心功能
   - 符合 OpenClaw 架构设计
   - 遵循 OpenClaw 开发规范

**输出：**
- 审核通过 → 任务完成
- 审核不通过 → 回退到开发节点

## 🔄 流转规则

### 正常流程

```
开发完成 → 提交验收 → 验收通过 → 提交审核 → 审核通过 → 任务完成
```

### 驳回流程（重点！）

```
审核不通过 → 回退到开发节点 → 重新开发 → 重新验收 → 重新审核
```

**关键点：**
1. ❌ **不是回退到上一个节点**（验收节点）
2. ✅ **直接回退到开发节点**
3. ✅ **需要重新走完整流程**（开发→验收→审核）

## 📊 状态机

| 当前状态 | 可流转到的状态 | 触发条件 |
|---------|--------------|---------|
| **开发中** | 验收中 | 开发者提交验收 |
| **验收中** | 审核中 | 项目负责人提交审核 |
| **验收中** | 开发中 | 验收不通过（可选） |
| **审核中** | 已完成 | 审核通过 |
| **审核中** | 开发中 | 审核不通过（重点！） |

---

## 🎯 审核不通过处理

### 回退机制

**审核不通过时：**

1. **回退目标：** 开发节点（不是验收节点！）
2. **重新执行：** 开发 → 验收 → 审核（完整流程）
3. **原因记录：** 审核意见必须明确记录

### 回退原因示例

**代码合规问题：**
- ❌ 代码不符合规范
- ❌ 缺少错误处理
- ❌ 硬编码配置

**系统稳定问题：**
- ❌ 影响系统稳定性
- ❌ 存在内存泄漏
- ❌ 性能问题严重

**OpenClaw 兼容问题：**
- ❌ 影响 OpenClaw 核心功能
- ❌ 不符合架构设计
- ❌ 违反开发规范

### 重新开发要求

**开发者收到驳回后：**

1. ✅ 仔细阅读审核意见
2. ✅ 修改代码问题
3. ✅ 重新自测
4. ✅ 提交重新验收
5. ✅ 重新走完整流程

## 📝 API 使用

### 创建开发任务

```bash
POST /api/dev-workflow/tasks

{
  "title": "BCE 任务管理模块开发",
  "description": "实现任务创建、分配、流转功能",
  "developer": "匠心",
  "projectOwner": "天枢"
}
```

### 开发完成，提交验收

```bash
POST /api/dev-workflow/tasks/:id/submit-acceptance

{
  "developer": "匠心",
  "deliverables": [
    "源代码链接",
    "自测报告",
    "开发文档"
  ]
}
```

### 验收通过，提交审核

```bash
POST /api/dev-workflow/tasks/:id/submit-audit

{
  "projectOwner": "天枢",
  "acceptanceReport": "功能符合需求，验收通过"
}
```

### 审核通过

```bash
POST /api/dev-workflow/tasks/:id/audit-pass

{
  "auditor": "执矩",
  "auditComments": "代码合规，系统稳定，审核通过"
}
```

### 审核不通过（回退到开发）

```bash
POST /api/dev-workflow/tasks/:id/audit-reject

{
  "auditor": "执矩",
  "rejectReason": "代码存在硬编码配置，影响系统稳定性，需要修改"
}

响应：
{
  "success": true,
  "message": "❌ 审核不通过，已回退到开发节点，需要重新开发 → 重新验收 → 重新审核",
  "workflow": {
    "currentNode": "development",
    "rollbackTo": "development",
    "reflowRequired": true,
    "nextSteps": ["重新开发", "重新验收", "重新审核"]
  }
}
```

## 🚨 注意事项

### 重点强调

1. **审核不通过回退到开发节点**
   - ❌ 不是验收节点
   - ✅ 是开发节点
   - ✅ 需要重新走完整流程

2. **执矩的审核权限**
   - ✅ 一票否决权
   - ✅ 可以直接回退到开发
   - ✅ 不需要经过验收同意

3. **重新开发的要求**
   - ✅ 必须修改审核指出的问题
   - ✅ 必须重新自测
   - ✅ 必须重新走完整流程

### 常见错误

1. ❌ 审核不通过只修改文档
   - ✅ 必须修改代码

2. ❌ 审核不通过直接提交审核
   - ✅ 必须先验收

3. ❌ 忽略审核意见
   - ✅ 必须逐条修改
