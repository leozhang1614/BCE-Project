# 第一部分：项目概述

## 1. 项目是什么？

### 1.1 一句话介绍

**BCE（北斗协同引擎）是一个任务管理和协同系统，帮助团队高效分配任务、跟踪进度、确保任务按时完成。**

### 1.2 形象比喻

想象你有一个**智能任务管家**：

```
你（管理者）说："匠心，去做 BCE 技术方案"
           ↓
任务管家自动：
  1. ✅ 记录这个任务
  2. ✅ 通知匠心
  3. ✅ 提醒匠心确认
  4. ✅ 跟踪任务进度
  5. ✅ 完成后通知你
```

**BCE 就是这个"任务管家"！**

### 1.3 核心能力

| 能力 | 说明 | 例子 |
|------|------|------|
| **任务分配** | 把任务分配给具体的人 | 天枢分配任务给匠心 |
| **及时通知** | 任务变动及时通知相关人员 | 匠心收到飞书通知 |
| **进度跟踪** | 实时查看任务做到哪一步了 | 查看任务看板 |
| **任务流转** | 任务在不同人之间传递 | 匠心→司库→执矩 |
| **完成确认** | 确保任务被确认和执行 | 匠心点击"确认收到" |

## 2. 为什么要做这个项目？

### 2.1 遇到的问题

**场景 1：任务发出去，没人理**
```
天枢："匠心，做这个任务"
匠心：（没看到消息）
天枢：（等了好久）"怎么还没开始？"
匠心："啊？我没收到通知..."
```
❌ **问题：** 任务通知不到位

**场景 2：任务做完了，没人知道**
```
匠心：（默默完成任务）
天枢：（不知道已完成）"任务怎么还没好？"
匠心："我早就做完了啊..."
```
❌ **问题：** 任务状态不透明

**场景 3：多人协作，责任不清**
```
天枢："这个任务谁负责？"
匠心："我以为司库在做"
司库："我以为执矩在处理"
```
❌ **问题：** 任务流转不清晰

### 2.2 BCE 的解决方案

**针对场景 1：三重通知保障**
```
天枢创建任务
     ↓
OCC 协作页面显示 ✅
     ↓
BCE 系统：
  1. 飞书卡片通知（即时）✅
  2. SSE 实时推送（替代轮询）✅
  3. 邮箱机制（可靠存储）✅
```
✅ **结果：** 匠心一定能收到通知

**针对场景 2：实时状态同步**
```
匠心完成任务 → 点击 [提交验收]
                    ↓
              OCC 任务页面立即更新
                    ↓
              天枢看到"待验收"状态
```
✅ **结果：** 所有人实时看到任务状态

**针对场景 3：流转历史追踪**
```
OCC 协作页面显示：
  09:00 天枢 → 匠心（任务分配）✅
  09:30 匠心确认接收 ✅
  10:00 匠心 → 司库（协作请求）✅
  10:30 司库提供数据 ✅
```
✅ **结果：** 谁在什么时候做了什么，一目了然

## 6. 详细功能说明

### 6.1 任务创建

**谁用：** 天枢（管理者）  
**什么时候用：** 有新工作需要分配时

**使用步骤：**

**方式 A：通过 OpenClaw 创建（推荐）**
```javascript
await sessions_send({
  sessionKey: 'agent-jiangxin',
  message: `
📋 新任务分配

【任务名称】BCE 技术方案设计
【优先级】P0
【截止时间】今天 18:00
【任务描述】设计 BCE 项目技术架构
`
});
```

**方式 B：通过 OCC 页面创建**
```
1. 访问 OCC 任务页面
   http://192.168.31.187:4310/?section=tasks
2. 点击 [新建任务] 按钮
3. 填写任务信息
4. 点击 [提交]
```

**方式 C：通过 API 创建**
```bash
curl -X POST http://localhost:3000/api/bce/tasks \
  -H "Content-Type: application/json" \
  -d '{
    "title": "BCE 技术方案设计",
    "assignee": "匠心",
    "priority": "P0"
  }'
```

### 6.2 任务确认

**谁用：** 匠心（任务负责人）  
**什么时候用：** 收到新任务通知时

**使用步骤：**

**方式 A：OCC 页面确认（推荐）**
```
1. 访问 OCC 协作页面
   http://192.168.31.187:4310/?section=collaboration
2. 看到待确认任务
3. 点击 [确认接收] 按钮
4. （可选）填写备注
5. 点击 [提交]
```

**方式 B：飞书卡片确认（待开发）**
```
1. 收到飞书卡片通知
   ┌─────────────────────────────┐
   │ 📋 新任务分配               │
   │ 任务：BCE 技术方案设计       │
   │                             │
   │ [✅ 确认接收] [❌ 驳回]     │
   └─────────────────────────────┘
2. 点击 [✅ 确认接收] 按钮
3. BCE 解析卡片回调并确认任务
```

**方式 C：飞书回复确认（待优化）**
```
1. 收到飞书通知
2. 回复"收到"（在通知消息下回复）
3. BCE 通过消息引用匹配确认任务
```

**方式 D：邮箱机制确认（新增）**
```
1. BCE 将任务写入邮箱
   runtime/inboxes/jiangxin/msg-001.json
2. 匠心查看邮箱
3. 点击 [确认接收]
4. 消息标记为已读
```

### 6.3 任务流转

```
天枢 → 匠心 → 司库 → 执矩 → 天枢
创建    执行    协作    审核    验收
```

```bash
curl -X POST http://localhost:3000/api/bce/tasks/:id/complete-and-transfer \
```

## 12. 详细实现说明

### 12.1 文件系统邮箱（v3.0 新增）

**目标：** 可靠消息存储，零依赖

**实现原理：**
```
runtime/inboxes/{agent}/msg-*.json

├── jiangxin/
│   ├── msg-001.json  ← 未读消息
│   ├── msg-002.json  ← 未读消息
│   └── read/
│       └── msg-000.json  ← 已读消息
├── siku/
└── ...
```

**消息格式：**
```json
{
  "id": "msg-001",
  "from": "tianshu",
  "to": "jiangxin",
  "type": "task_assigned",
  "taskId": "TASK-001",
  "content": "📋 新任务分配：BCE 技术方案设计",
  "createdAt": "2026-03-21T10:00:00.000Z",
  "read": false,
  "confirmedAt": null
}
```

**核心代码：**
```javascript
class MailboxService {
  constructor() {
    this.mailboxDir = 'runtime/inboxes';
  }

  // 发送消息到邮箱
  async send(to, from, type, content, taskId) {
    const agentDir = path.join(this.mailboxDir, to);
    await fs.ensureDir(agentDir);
    
    const msg = {
      id: `msg-${Date.now()}`,
      from,
      to,
      type,
      taskId,
      content,
      createdAt: new Date().toISOString(),
      read: false
    };
    
    // 原子写入（tmp + rename）
    const tmpFile = path.join(agentDir, `.tmp-${msg.id}.json`);
    const msgFile = path.join(agentDir, `${msg.id}.json`);
    
    await fs.writeFile(tmpFile, JSON.stringify(msg, null, 2));
    await fs.rename(tmpFile, msgFile);
    
    return msg;
  }

  // 接收消息
  async receive(agent, consume = true) {
    const agentDir = path.join(this.mailboxDir, agent);
    const files = await fs.readdir(agentDir);
    const msgs = [];
    
    for (const file of files) {
      if (file.endsWith('.json')) {
        const content = await fs.readFile(path.join(agentDir, file));
        msgs.push(JSON.parse(content));
        
        if (consume) {
          // 移动到已读目录
          const readDir = path.join(agentDir, 'read');
          await fs.ensureDir(readDir);
          await fs.rename(
            path.join(agentDir, file),
            path.join(readDir, file)
          );
        }
      }
    }
    
    return msgs;
  }

  // 未读计数
  async count(agent) {
    const agentDir = path.join(this.mailboxDir, agent);
    const files = await fs.readdir(agentDir);
    return files.filter(f => f.endsWith('.json')).length;
  }
}
```

### 12.2 飞书 Webhook 优化（v3.0 更新）

**目标：** 用户在飞书群回复"收到"后，系统自动确认任务

**核心改进：** 使用消息引用匹配，避免歧义

**实现原理：**
```
1. 用户在飞书群回复消息（引用原通知）
   ↓
2. 飞书服务器发送 HTTP POST 到 BCE Webhook
   ↓
3. BCE 接收并解析消息内容
   ↓
4. 提取被引用的原消息 ID（parent_id）
   ↓
5. 根据原消息 ID 定位任务
   ↓
6. 调用 OCC API 确认任务
   ↓
7. 回复确认结果到飞书群
```

**核心代码：**
```javascript
// 从飞书事件中提取被引用的消息 ID
const parentMessageId = event.message.parent_id;

// 根据原消息 ID 定位任务
```

### 12.4 SSE 实时推送（v3.0 新增）

**目标：** 替代定时轮询，实时推送任务更新

**实现原理：**
```
浏览器/客户端 ─── SSE 连接 ───> BCE 服务器
                                  │
                                  ↓
                            监听 OCC 变化
                                  │
                                  ↓
                            推送任务更新
```

**核心代码：**
```javascript
const express = require('express');
const { Response } = require('sse.js');

router.get('/sse/tasks', async (req, res) => {
  const sse = new Response(req, res);
  
  // 定期推送心跳
  const heartbeat = setInterval(() => {
    sse.send({ event: 'heartbeat', data: Date.now() });
  }, 30000);
  
  // 监听 OCC 任务变化
  const watcher = watchOCCTasks(async (task) => {
    sse.send({ event: 'task_update', data: task });
  });
  
  // 客户端断开时清理
  res.on('close', () => {
    clearInterval(heartbeat);
    watcher.stop();
  });
});
```

**客户端使用：**
```javascript
const eventSource = new EventSource('http://localhost:3000/sse/tasks');

  console.log('心跳:', event.data);
```

# 第六部分：架构演进

## 16. 当前方案局限性

### 16.1 实时性问题

**问题：** 定时轮询导致最多 5 分钟延迟

**影响：**
- 任务通知不及时
- 状态同步延迟

**解决方案（v3.0）：**
- ✅ SSE 实时推送替代轮询
- ✅ 毫秒级推送

### 16.2 数据一致性问题

**问题：** OCC 和 BCE 可能存在状态不一致

**影响：**
- 基于过期状态发送提醒
- 任务流转错误

**解决方案（v3.0）：**
- ✅ 确立 OCC 为唯一数据源
- ✅ BCE 所有写操作通过 OCC API 执行
- ✅ 冲突时以 OCC 状态为准

### 16.3 交互体验问题

**问题：** 飞书纯文本通知体验差

**影响：**
- 确认操作繁琐
- 容易误操作

**解决方案（v3.0）：**
- ✅ 使用交互式卡片（带按钮）
- ✅ 消息引用匹配避免歧义
- ✅ 邮箱机制可靠存储

### 16.4 可靠性问题

**问题：** 轮询可能遗漏任务

**影响：**
- 任务通知不到位
- 用户收不到提醒

**解决方案（v3.0）：**
- ✅ 文件系统邮箱机制
- ✅ 原子写入（tmp + rename）
- ✅ 消息持久化存储

## 19. BCE 改进方案

### 19.1 基础交互功能提升对比

| 功能 | v2.0 | v3.0 | 提升 |
|------|------|------|------|
| **通知方式** | 定时轮询（5 分钟） | SSE 实时推送（毫秒级） | ⬆️ 60 倍 |
| **消息存储** | 内存缓存 | 文件系统邮箱 | ⬆️ 可靠持久化 |
| **确认方式** | 飞书文本回复 | 飞书卡片按钮 | ⬆️ 零歧义 |
| **任务 ID** | 自增 ID | UUID | ⬆️ 全局唯一 |
| **消息匹配** | 关键词匹配 | 消息引用匹配 | ⬆️ 100% 准确 |
| **通知上限** | 无限制 | 最多 3 次 | ⬆️ 避免骚扰 |
| **安静时段** | 无 | 23:00-07:00 | ⬆️ 人性化 |

### 19.2 核心指标提升

| 指标 | v2.0 | v3.0 | 提升幅度 |
|------|------|------|---------|
| **通知延迟** | 最多 5 分钟 | 毫秒级 | ⬆️ 30000 倍 |
| **确认准确率** | ~80% | ~100% | ⬆️ 20% |
| **消息可靠性** | 内存缓存 | 文件持久化 | ⬆️ 100% |
| **用户体验** | 文本回复 | 卡片按钮 | ⬆️ 显著 |
| **系统依赖** | 轮询服务 | SSE+ 邮箱 | ⬆️ 更简单 |

### 19.3 架构优化

**v2.0 架构：**
```
定时轮询（5 分钟）→ 扫描 OCC → 飞书通知
```

**v3.0 架构：**
```
OCC 变化 → SSE 实时推送 → 飞书卡片
          ↓
      文件系统邮箱（可靠存储）
```

**优势：**
- ✅ 实时性：毫秒级 vs 5 分钟
- ✅ 可靠性：文件持久化 vs 内存缓存
- ✅ 准确性：消息引用匹配 vs 关键词匹配
- ✅ 体验：卡片按钮 vs 文本回复
