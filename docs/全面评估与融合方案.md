## 📊 现状分析

### 当前 BCE 方案

**架构：**
```
用户 → BCE API → 飞书机器人 API → 飞书群 → 用户
```

**功能：**
- ✅ 任务创建、分配、执行、验收流程
- ✅ 飞书@人通知
- ✅ 任务状态管理
- ✅ 持久化存储（runtime/bce-data.json）

**问题：**
- ❌ 飞书@通知失效（权限/user_id/限流）
- ❌ 依赖外部 API（不稳定）
- ❌ 信息孤岛（与 OpenClaw 原生数据不互通）
- ❌ 缺少可视化界面（只有简单 HTML）

---

### OpenClaw Control Center 原生能力

**架构：**
```
OpenClaw Gateway → OCC 适配器 → OCC UI → 用户
```

**核心功能：**

#### 1. 协作页面（Collaboration）
- ✅ **跨会话消息追踪** - 记录所有 Agent 之间的消息传递
- ✅ **父子任务关联** - 显示任务流转关系
- ✅ **实时数据源** - 直接读取 OpenClaw 会话数据
- ✅ **可视化展示** - 专业 UI 界面

**代码位置：**
```
src/runtime/collaboration.ts
src/ui/pages/Collaboration.tsx
```

**数据流：**
```typescript
// 从 OpenClaw 读取会话数据
const sessions = await adapter.getSessions();

// 提取跨会话消息
const interSessionMessages = sessions.flatMap(s => 
  s.messages.filter(m => m.tool === 'sessions_send')
);

// 构建协作关系图
const collaborationGraph = buildCollaborationGraph(interSessionMessages);
```

#### 2. 任务页面（Tasks）
- ✅ **任务板** - 可视化任务状态
- ✅ **审批流** - 支持审批确认
- ✅ **执行链** - 追踪任务执行历史
- ✅ **运行时证据** - 关联实际执行记录

#### 3. 通知中心（Notification）
- ✅ **待确认事项** - 集中展示需要确认的任务
- ✅ **审批请求** - 支持审批操作
- ✅ **审计日志** - 记录所有操作

#### 4. 员工页面（Staff）
- ✅ **实时状态** - 显示每个 Agent 的工作状态
- ✅ **任务队列** - 显示待办任务
- ✅ **最近输出** - 显示最新工作成果

| 维度 | BCE 方案 | OCC 原生方案 |
|------|---------|-------------|

| **扩展性** | 有限 | 强（持续更新） |

## 💡 融合方案评估

### 方案 A：完全放弃 BCE，使用 OCC 原生

**实施：**
1. 停用 BCE 系统
2. 部署 OCC（`npm run dev:ui`）
3. 使用 OCC 协作页面 + 任务页面
4. 通过 `sessions_send` 实现 Agent 间消息传递

**优势：**
- ✅ 稳定可靠（不依赖外部 API）
- ✅ 功能完整（协作、任务、通知、审批）
- ✅ 数据统一（直接读取 OpenClaw）
- ✅ 维护简单（单一系统）
- ✅ 持续更新（GitHub 原项目）

**劣势：**
- ❌ 需要学习成本（新界面）
- ❌ 没有飞书@通知（可以接受）
- ❌ 需要主动查看页面（可以培养习惯）

**适用场景：**
- 团队愿意使用 Web 界面
- 不依赖飞书@功能
- 追求稳定性和可维护性

---

### 方案 B：BCE 与 OCC 深度融合

**架构：**
```
用户 → BCE API → OCC 协作服务 → OpenClaw Gateway
     ↓
  飞书 Webhook（可选）
```

**融合点：**

#### 1. 数据层融合
```javascript
// BCE 任务同步到 OCC
await occAdapter.syncTask({
  id: bceTask.id,
  title: bceTask.title,
  status: mapBceStatusToOcc(bceTask.status),
  assignee: bceTask.assignee,
  sessionKey: `bce-${bceTask.id}`
});
```

#### 2. 通知层融合
```javascript
// BCE 触发 OCC 通知
await occAdapter.notify({
  type: 'task_assigned',
  taskId: bceTask.id,
  targetAgent: bceTask.assignee
});

// 同时发送飞书消息（可选）
await feishuNotify.send(bceTask);
```

#### 3. 协作层融合
```javascript
// BCE 使用 OCC 协作页面展示
// URL: http://localhost:4310/?section=collaboration&lang=zh
// 显示 BCE 任务流转记录
```

**优势：**
- ✅ 保留 BCE 的任务管理逻辑
- ✅ 利用 OCC 的协作和展示能力
- ✅ 飞书通知作为可选辅助
- ✅ 渐进式迁移，风险低

**劣势：**
- ⚠️ 系统复杂度高（双重系统）
- ⚠️ 数据同步可能不一致
- ⚠️ 维护成本高

### 方案 C：OCC 为主 + BCE 为辅（推荐）

**架构：**

## 📋 最终推荐：方案 C（OCC 为主 + BCE 为辅）

### 理由

1. **稳定性优先**
   - OCC 直接读取 OpenClaw 数据，不依赖外部 API
   - 本地部署，完全可控

2. **功能完整**
   - 协作、任务、通知、审批、审计
   - 专业 UI，可视化强

3. **渐进式迁移**
   - 保留 BCE 飞书通知作为辅助
   - 降低团队适应成本
   - 风险可控

4. **长期维护**
   - OCC 是 OpenClaw 官方项目
   - 持续更新，社区支持
   - BCE 可作为定制扩展
