# BCE 任务交互问题解决方案

**日期：** 2026-03-21 08:30  
**提出人：** 匠心 (CTO)  
**背景：** 飞书@通知失效，任务发出后对方收不到

---

## 🔍 问题分析

### 当前方案（飞书机器人通知）
- ❌ 飞书应用权限配置复杂
- ❌ user_id 每个应用独立，容易出错
- ❌ 机器人可能被限流或屏蔽
- ❌ 依赖外部 API，不稳定

### 测试结果
- ✅ BCE 系统正常发送通知
- ✅ 飞书 API 返回成功
- ❌ **群里成员收不到@通知**

---

## 💡 解决方案

### 方案 1：OpenClaw Control Center 协作页面（推荐）

**核心思路：** 使用 OpenClaw 原生的协作页面功能，替代飞书通知

**功能特点：**
1. ✅ **实时任务板** - 所有任务可视化展示
2. ✅ **状态追踪** - 任务状态实时更新
3. ✅ **跨会话信号** - 记录所有交互历史
4. ✅ **无需飞书** - 不依赖外部 API
5. ✅ **内置集成** - 直接读取 OpenClaw 数据

**实施步骤：**
1. 部署 OpenClaw Control Center（原版）
2. 配置与 OpenClaw Gateway 的连接
3. 使用协作页面查看任务
4. 通过页面进行任务确认和反馈

**访问地址：** `http://localhost:4310/?section=collaboration&lang=zh`

**优势：**
- 稳定可靠，不依赖第三方
- 功能完整，有任务板、协作视图
- 直接集成 OpenClaw，数据实时同步

**劣势：**
- 需要主动打开页面查看
- 没有推送通知（可以配合浏览器通知）

---

### 方案 2：飞书群机器人 Webhook（简化版）

**核心思路：** 使用飞书群机器人的简单 Webhook，不@人，只发消息

**实施步骤：**
1. 在飞书群添加"自定义机器人"
2. 获取 Webhook URL
3. BCE 系统调用 Webhook 发送消息
4. 不@人，只发普通消息

**优势：**
- 配置简单，不需要 AppID/AppSecret
- 不需要 user_id 映射
- 消息一定能发送到群里

**劣势：**
- 不能@人
- 功能有限

---

### 方案 3：多通道通知（混合方案）

**核心思路：** 同时使用多种通知方式，确保至少一种有效

**通知通道：**
1. ✅ 飞书机器人@通知（当前方案，但有局限）
2. ✅ OpenClaw Control Center 协作页面（主动查看）
3. ✅ 飞书群机器人 Webhook（普通消息）
4. ⏳ 邮件通知（可选）
5. ⏳ 短信通知（可选）

**实施策略：**
- 重要任务：所有通道同时通知
- 普通任务：协作页面 + Webhook
- 紧急任务：飞书@ + 短信

---

## 🎯 推荐方案：方案 1 + 方案 3 结合

### 第一阶段（立即实施）
1. **部署 OpenClaw Control Center**
   - 使用原版项目：https://github.com/TianyiDataScience/openclaw-control-center
   - 配置连接 OpenClaw Gateway
   - 启用协作页面

2. **保留飞书通知作为辅助**
   - 继续发送飞书消息（即使没有@）
   - 作为提醒手段

3. **建立任务查看习惯**
   - 团队成员每天查看协作页面
   - 晨会/夕会检查任务状态

### 第二阶段（优化改进）
1. **添加浏览器通知**
   - 协作页面支持浏览器推送通知
   - 任务更新时弹窗提醒

2. **集成更多通知通道**
   - 邮件通知
   - 短信通知（紧急情况）

3. **自动化任务流转**
   - 任务超时自动升级
   - 自动分配任务
