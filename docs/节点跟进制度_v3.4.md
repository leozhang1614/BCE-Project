## 制度总则

### 第一条 目的

为规范公司项目节点管理，确保任务按时高质量交付，特制定本制度。

### 第二条 适用范围

本制度适用于公司所有项目任务，包括但不限于：

- 技术开发任务
- 产品设计任务
- 运营推广任务
- 职能支持任务

### 第三条 核心原则

1. **进度透明** - 所有任务进度实时可见
2. **强制反馈** - 系统强制，不依赖自觉
3. **实质权力** - 管理者有真正的干预权力
4. **自动升级** - 预警自动升级，不依赖人工
5. **绩效绑定** - 进度直接计入绩效

## 节点定义

### 第四条 节点类型

公司任务节点分为以下类型：

#### 1. 里程碑节点（Milestone）

**定义：** 项目关键阶段的完成点

**特征：**
- 有明确交付物
- 需要验收
- 影响后续工作

**示例：**
- 财务知识库搭建完成
- 智能决策模块上线
- 前端界面设计定稿

---

#### 2. 进度节点（Progress）

**定义：** 任务执行过程中的进度检查点

**特征：**
- 按时间周期设置
- 需要汇报进度百分比
- 用于过程监控

**周期：**
- 标准任务：30 分钟
- 紧急任务（P0）：15 分钟
- 长期任务：1 小时

---

#### 3. 汇报节点（Report）

**定义：** 标准化进度汇报时间点

**特征：**
- 固定时间推送
- 按模板格式回复
- 全员统一执行

**时间表：**

| 时间 | 类型 | 说明 |
|------|------|------|
| 09:00 | 早间汇报 | 当日工作计划 |
| 10:30 | 定时汇报 | 上午进度更新 |
| 12:00 | 午间汇报 | 上午工作总结 |
| 13:30 | 定时汇报 | 下午进度更新 |
| 15:00 | 定时汇报 | 下午进度更新 |
| 16:30 | 定时汇报 | 傍晚进度更新 |
| 18:00 | 晚间汇报 | 当日工作总结 |
| 19:30 | 最终汇报 | 最终进度确认 |

---

#### 4. 验收节点（Acceptance）

**定义：** 任务完成后的验收确认点

**特征：**
- 执行者提交完成
- 验收人审核通过
- 任务状态变更为"已验收"

**流程：**
```
执行者提交 → 验收人审核 → (通过) 任务完成
                        → (驳回) 返回修改
```

---

### 第五条 节点优先级

节点按优先级分为三级：

| 优先级 | 标识 | 说明 | 跟进频率 |
|--------|------|------|----------|
| P0 | 🔴 | 紧急重要 | 15 分钟 |
| P1 | 🟠 | 重要 | 30 分钟 |
| P2 | 🟡 | 常规 | 1 小时 |

**优先级调整：**
- 管理者可根据业务需要调整
- P0 任务需 CEO 批准
- 调整后立即通知执行者

## 跟进机制

### 第六条 进度反馈机制

#### 1. 强制进度反馈

**要求：**
- 执行者确认任务后，系统启动计时
- 每 30 分钟必须更新一次进度百分比
- 更新内容包括：
  - 完成百分比（0-100）
  - 已完成工作内容
  - 剩余工作内容
  - 预计完成时间
  - 风险/阻塞（如有）

**更新方式：**
- 飞书卡片（推荐）
- API 接口（技术人员）

**豁免情况：**
- 执行者主动报告困难
- 管理者批准延期
- 任务被重新分配

---

#### 2. 标准化汇报

**要求：**
- 系统每 1.5 小时推送汇报模板
- 执行者 30 分钟内按格式回复
- 汇报内容真实准确

**汇报模板：**

```
📋 进度汇报

任务：{task_title}
负责人：{assignee}
截止时间：{deadline}
当前状态：{status}

━━━━━━━━━━━━━━━━━━━━

✅ 完成百分比：60%

✅ 已完成：
   - 数据采集模块
   - 数据清洗模块

✅ 剩余工作：
   - 三维分析
   - 建议生成

✅ 预计完成：2026-03-23 23:00

✅ 风险/阻塞：无

━━━━━━━━━━━━━━━━━━━━
```

### 第七条 预警升级机制

#### 1. 预警级别

| 级别 | 触发条件 | 通知对象 | 影响 |
|------|----------|----------|------|
| 🟡 一级预警 | 30 分钟未更新 | 执行者 | 提醒 |
| 🟠 二级预警 | 60 分钟未更新 | 管理者 | 建议介入 |
| 🔴 三级预警 | 120 分钟未更新 | CEO | 标记高风险 |

---

#### 2. 预警流程

```
┌─────────────────────────────────────────────────────────┐
│  预警升级流程                                            │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  0'          30'          60'          120'            │
│  │            │            │             │              │
│  │            ▼            ▼             ▼              │
│  │        提醒执行者   通知管理者    上报 CEO          │
│  │            │            │             │              │
│  │            │            │             │              │
│  │        记录 missed   建议介入      标记风险        │
│  │        update=1      帮助/重分配   计入绩效        │
│  │                                                         │
└─────────────────────────────────────────────────────────┘
```

---

#### 3. 预警解除

**条件：**
- 执行者更新进度
- 进度百分比有提升
- 系统自动解除预警

**流程：**
```
执行者更新进度 → 系统检测到进度提升 → 重置预警标记 → 计时器重新开始
```

### 第八条 管理者干预机制

#### 1. 干预权限

管理者拥有以下实质性权力：

| 权限 | 说明 | 适用场景 |
|------|------|----------|
| 重新分配 | 更换执行者 | 进度滞后、能力不足 |
| 调整优先级 | P0/P1/P2 变更 | 业务需求变化 |
| 调整截止时间 | 延期或提前 | 业务需求调整 |
| 强制汇报 | 要求立即更新 | 紧急情况 |
| 标记风险 | 高亮问题任务 | 进度滞后、技术难点 |
| 上报 CEO | 需要决策 | 资源不足、范围调整 |

---

#### 2. 干预流程

```
管理者发现风险 → 选择干预方式 → 系统执行 → 通知相关人员 → 记录历史
```

**示例：**

```
场景：磐石负责的"数据感知模块"进度滞后

1. 司库（管理者）查看仪表盘，发现进度 30%，滞后 50%
2. 司库决定重新分配任务给匠心
3. 系统执行重新分配：
   - 更新任务执行者
   - 通知磐石（绩效 -15 分）
   - 通知匠心（新任务分配）
   - 通知天枢（备案）
4. 记录到任务历史
```

---

#### 3. 干预原则

**最佳实践：**
1. **主动介入** - 不要等待预警升级
2. **提供帮助** - 而不是指责
3. **记录决策** - 便于事后复盘
4. **及时沟通** - 与执行者和 CEO 保持沟通

### 第九条 绩效绑定机制

#### 1. 绩效计算规则

**基础分（准时性）：**

| 情况 | 分数 | 说明 |
|------|------|------|
| 准时完成 | +10 | 在截止时间前完成 |
| 提前 2 小时以上 | +20 | 额外奖励 |
| 延期完成 | -10 | 基础扣分 |
| 延期每小时 | -5 | 累计扣分 |

**过程分（进度汇报）：**

| 情况 | 分数 | 说明 |
|------|------|------|
| 每次主动汇报 | +2 | 鼓励及时更新 |
| 每次未汇报 | -5 | 惩罚不更新 |

**影响分（任务调整）：**

| 情况 | 分数 | 说明 |
|------|------|------|
| 被重新分配 | -15 | 执行者原因 |
| 高风险标记 | -10 | 严重滞后 |

**质量分（验收评分）：**

| 验收评分 | 分数 | 说明 |
|----------|------|------|
| 5 分（优秀） | +10 | 超出预期 |
| 4 分（良好） | +5 | 达到预期 |
| 3 分（合格） | 0 | 基本合格 |
| 2 分（需改进） | -5 | 未达预期 |
| 1 分（不合格） | -10 | 严重不达标 |

---

#### 2. 绩效等级

| 总分 | 等级 | 说明 | 奖金系数 |
|------|------|------|----------|
| ≥50 | S | 卓越表现 | 1.2 |
| 30-49 | A | 优秀表现 | 1.1 |
| 10-29 | B | 良好表现 | 1.0 |
| 0-9 | C | 需要改进 | 0.9 |
| <0 | D | 严重不达标 | 0.7 |

---

#### 3. 绩效应用

**S 级（卓越）：**
- 奖金 +20%
- 优先分配重要任务
- 月度优秀员工候选

**A 级（优秀）：**
- 奖金 +10%
- 优先选择任务

**B 级（良好）：**
- 正常奖金
- 正常分配

**C 级（需改进）：**
- 奖金 -10%
- 口头警告
- 绩效面谈

**D 级（不达标）：**
- 奖金 -30%
- 书面警告
- 绩效改进计划
- 可能调岗

## 职责分工

### 第十条 执行者职责

**主要职责：**
1. 确认任务（30 分钟内）
2. 更新进度（每 30 分钟）
3. 按模板汇报（每 1.5 小时）
4. 遇到困难立即报告
5. 完成任务提交验收

**工作要求：**
- 进度真实准确
- 汇报及时规范
- 沟通主动积极
- 交付按时高质量

---

### 第十一条 管理者职责

**主要职责：**
1. 查看仪表盘（每天早中晚）
2. 识别风险任务
3. 介入帮助执行者
4. 必要时重新分配
5. 上报 CEO 决策

**工作要求：**
- 主动介入，不等待
- 提供帮助，不指责
- 记录决策，便复盘
- 沟通及时，信息透明

---

### 第十二条 CEO 职责

**主要职责：**
1. 查看全公司仪表盘
2. 审批重大决策
3. 接收高风险上报
4. 查看绩效报告
5. 优化资源配置

**工作要求：**
- 数据驱动决策
- 快速响应上报
- 平衡各方利益
- 记录决策历史

---

### 第十三条 验收人职责

**主要职责：**
1. 审核任务完成质量
2. 给出验收评分
3. 提供改进建议
4. 确认任务完成

**工作要求：**
- 客观公正
- 标准统一
- 反馈及时
- 记录详细

## 考核标准

### 第十四条 进度考核

**考核指标：**

| 指标 | 目标值 | 权重 | 说明 |
|------|--------|------|------|
| 进度更新及时率 | ≥95% | 30% | 30 分钟内更新比例 |
| 汇报回复及时率 | ≥90% | 25% | 30 分钟内回复比例 |
| 任务准时完成率 | ≥85% | 25% | 截止时间前完成比例 |
| 预警触发次数 | ≤2 次/月 | 10% | 越少越好 |
| 重新分配次数 | ≤1 次/月 | 10% | 越少越好 |

---

### 第十五条 质量考核

| 指标 | 目标值 | 权重 | 说明 |

| 验收通过率 | ≥90% | 30% | 一次验收通过比例 |
| 验收平均分 | ≥4.0 | 25% | 5 分制 |
| 返工次数 | ≤1 次/月 | 20% | 越少越好 |
| 文档完整率 | 100% | 15% | 必需有文档 |
| 代码规范符合率 | ≥95% | 10% | 符合代码规范 |

---

### 第十六条 沟通考核

**考核指标：**

| 指标 | 目标值 | 权重 | 说明 |
|------|--------|------|------|
| 困难报告及时率 | 100% | 40% | 遇到困难立即报告 |
| 响应速度 | ≤30 分钟 | 30% | 回复消息速度 |
| 协作评价 | ≥4.0 | 30% | 同事评价（5 分制） |

### 第十九条 申诉机制

**申诉条件：**
- 绩效计算错误
- 预警误触发
- 重新分配非执行者原因
- 其他不公正情况

**申诉流程：**
```
员工申诉 → 管理者初审 → CEO 终审 → 结果通知 → 执行调整
```

**申诉时效：**
- 月度绩效：次月 5 日前
- 季度绩效：次季度 10 日前
- 年度绩效：次年 15 日前
