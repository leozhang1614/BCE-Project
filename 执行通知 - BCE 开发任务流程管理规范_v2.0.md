## 二、开发任务流程（重点！）

### 适用范围
- ✅ 新功能开发
- ✅ Bug 修复
- ✅ 代码重构
- ✅ 性能优化

### 流程步骤

```
步骤 1：创建任务 → 步骤 2：开发 → 步骤 3：验收 → 步骤 4：审核 → 步骤 5：完成
```

#### 步骤 1：创建任务
**谁：** 项目负责人  
**怎么做：**
```
创建一个开发任务，名称是"用户模块开发"，开发者是匠心，负责人是天枢
```
**输出：** 任务 ID（如 dev-123）

#### 步骤 2：开发
**谁：** 开发者  
**怎么做：**
1. 确认任务（30 分钟内）
2. 编写代码
3. 自测通过
4. 编写交付物（源代码、自测报告、文档）
5. 提交验收

**时限：** 按任务 deadline（P0 今日 17:00）

**输出：**
```
提交任务 dev-123 的验收，交付物有源代码、自测报告、文档
```

#### 步骤 3：验收
**谁：** 项目负责人  
**怎么做：**
1. 验证功能
2. 检查需求符合度
3. 编写验收报告
4. 提交审核

**时限：** 开发完成后 2 小时内

**输出：**
```
任务 dev-123 验收通过，提交审核
```

#### 步骤 4：审核
**谁：** 执矩  
**怎么做：**
1. 检查代码合规性
2. 评估系统稳定性
3. 评估 OpenClaw 兼容性
4. 决定通过/驳回

**时限：** 验收通过后 1 小时内

**输出：**
- **通过：** `审核通过任务 dev-123，代码合规`
- **驳回：** `驳回任务 dev-123，原因：代码有硬编码`

#### 步骤 5：完成
**谁：** 系统自动  
**怎么做：**
- 审核通过 → 任务完成
- 审核驳回 → 回到步骤 2（重新开发）

```
4. 执矩：（审核后）审核通过任务 dev-123，代码合规
```

### 示例 2：修复 Bug

**场景：** 修复登录失败 Bug

**流程：**
```

4. 匠心：（修改后）重新提交验收

6. 执矩：（重新审核）审核通过任务 dev-125
```
