## 🎯 核心功能

### 标准流程
```
开发（开发者） → 验收（项目负责人） → 审核（执矩）
                                        ↓
                                 审核不通过
                                        ↓
                                 回退到开发节点
```

### 关键特性
1. ✅ **三节点固定流程** - 开发 → 验收 → 审核
2. ✅ **自动流转** - 节点完成后自动流转到下一节点
3. ✅ **审核不通过回退到开发** - 不是上一个节点，是开发节点
4. ✅ **重新走完整流程** - 重新开发 → 重新验收 → 重新审核

## 📋 适用范围

**适用任务：**
- ✅ 功能开发

## 🔧 使用方法

### 方式 1：BCE 系统 Web 界面

1. 访问：http://192.168.31.187:3000/bce-tasks.html
2. 创建任务时选择"开发任务"类型
3. 填写开发者、项目负责人
4. 系统自动启用标准工作流

### 方式 2：API 调用

**创建开发任务：**
```bash
POST http://localhost:3000/api/dev-workflow/tasks

{
  "title": "BCE 任务管理模块开发",
  "description": "实现任务创建、分配、流转功能",
  "developer": "匠心",
  "projectOwner": "天枢"
}
```

**开发完成提交验收：**
```bash
POST http://localhost:3000/api/dev-workflow/tasks/:id/submit-acceptance

{
  "developer": "匠心",
  "deliverables": ["源代码链接", "自测报告", "开发文档"]
}
```

**验收通过提交审核：**
```bash
POST http://localhost:3000/api/dev-workflow/tasks/:id/submit-audit

{
  "projectOwner": "天枢",
  "acceptanceReport": "功能符合需求，验收通过"
}
```

**审核通过：**
```bash
POST http://localhost:3000/api/dev-workflow/tasks/:id/audit-pass

{
  "auditor": "执矩",
  "auditComments": "代码合规，系统稳定，审核通过"
}
```

**审核不通过（回退到开发）：**
```bash
POST http://localhost:3000/api/dev-workflow/tasks/:id/audit-reject

{
  "auditor": "执矩",
  "rejectReason": "代码存在硬编码配置，需要修改"
}
```

## 📊 角色职责

### 开发者
- ✅ 功能实现
- ✅ 单元测试
- ✅ 自测通过
- ✅ 提交验收

### 项目负责人
- ✅ 功能验证
- ✅ 需求符合度检查
- ✅ 用户体验测试
- ✅ 提交审核

### 执矩（审核人）
- ✅ 代码合规性审核
- ✅ 系统稳定性评估
- ✅ OpenClaw 功能影响评估
- ✅ 审核通过/驳回

---

## ⏰ 时限要求

| 节点 | 时限 | 超时处理 |
|------|------|---------|
| **开发** | 按任务要求 | 系统提醒 |
| **验收** | 开发完成后 2 小时 | 通知开发者 |
| **审核** | 验收通过后 1 小时 | 通知项目负责人 |

---

## 🚨 审核标准

### 代码合规
- ✅ 符合代码规范
- ✅ 无硬编码配置
- ✅ 错误处理完整

### 系统稳定
- ✅ 不影响系统稳定性
- ✅ 无内存泄漏
- ✅ 性能良好

### OpenClaw 兼容
- ✅ 不影响核心功能
- ✅ 符合架构设计
- ✅ 遵循开发规范
