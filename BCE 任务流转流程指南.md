**版本：** v3.2

## 🔄 完整流程图

```
创建任务 → 发送通知 → 确认接收 → 执行任务 → 提交完成 → 自动流转 → 验收闭环
   ↓           ↓           ↓           ↓           ↓           ↓           ↓
 天枢       三通道      负责人      负责人      负责人      规则引擎    天枢
                                              ↓
                                          审核不通过
                                              ↓
                                        驳回回退 v3.3
                                              ↓
                                      回退到上一个执行节点
```

## 1️⃣ 任务创建（天枢/管理员）

### 方式一：OCC 面板创建（推荐）
```
1. 访问 http://192.168.31.187:4310/?section=tasks
2. 点击 [新建任务]
3. 填写信息：
   - 标题：清晰描述任务内容
   - 负责人：选择具体成员
   - 优先级：P0/P1/P2/P3
   - 截止时间：设置 deadline
4. 点击 [创建]
```

### 方式二：API 创建
```bash
curl -X POST http://localhost:3000/api/bce/tasks \
  -H "Content-Type: application/json" \
  -d '{
    "title": "任务标题",
    "description": "任务描述",
    "assignee": "负责人姓名",
    "priority": "P0",
    "dueDate": "2026-03-22T18:00:00Z"
  }'
```

### 方式三：Control Center 同步
```
在 Control Center 创建任务后，自动同步到 BCE 系统
```

优先级：P0
截止时间：2026-03-22 18:00

## 3️⃣ 任务确认（负责人）⚠️ **必须执行**

### 方式一：OCC 协作页面（推荐）
```
1. 访问 http://192.168.31.187:4310/?section=collaboration&lang=zh
2. 找到待确认任务
3. 点击 [确认接收] 按钮
4. 填写备注（可选）
5. 点击 [提交]
```

### 方式二：API 确认
```bash
curl -X POST http://localhost:3000/api/bce/tasks/{任务 ID}/confirm \
  -H "Content-Type: application/json" \
  -d '{"userName":"你的名字"}'
```

### 方式三：飞书回复
```
在飞书群回复："收到 + 姓名 + 任务类型"
例：收到，匠心，培训任务
```

### ⏰ 确认时效要求
| 优先级 | 确认时限 | 超时处理 |
|--------|---------|---------|
| **P0** | 30 分钟内 | 系统自动提醒 |
| **P1** | 2 小时内 | 系统自动提醒 |
| **P2** | 24 小时内 | 系统自动提醒 |
| **P3** | 72 小时内 | 系统自动提醒 |

### ⚠️ 超时升级机制
- **5 分钟**：系统发送第 2 次提醒
- **15 分钟**：通知任务创建人（天枢）
- **30 分钟**：标记为"超时未确认"，上报磊哥

## 4️⃣ 任务执行（负责人）

### 状态变更
```
待开始 → 进行中 → 已完成/阻塞/待验收
```

### 方式一：OCC 面板更新状态
```
1. 访问任务页面
2. 找到任务
3. 点击 [开始执行]
4. 完成后点击 [完成]
```

### 方式二：API 更新状态
```bash
# 开始执行
curl -X PUT http://localhost:3000/api/bce/tasks/{任务 ID} \
  -H "Content-Type: application/json" \
  -d '{"status":"executing","userName":"你的名字"}'

# 完成任务
curl -X POST http://localhost:3000/api/bce/tasks/{任务 ID}/complete \
  -H "Content-Type: application/json" \
  -d '{"userName":"你的名字","comment":"完成说明"}'
```

## 5️⃣ 任务完成（负责人）

### 完成操作
```
1. OCC 面板点击 [完成] 按钮
2. 或 API 提交完成
3. 填写完成说明（必填）
4. 提交验收
```

### 自动流转（规则引擎）
```
技术方案完成 → 自动转交司库（财务审核）
财务审核完成 → 自动转交执矩（安全审核）
安全审核完成 → 自动转交天枢（CEO 验收）
```

### 循环检测
```
系统自动检测循环转交，阻止死循环
例：匠心→司库→执矩→匠心（第 3 次拦截）
```

## 5.5️⃣ 任务驳回（审核节点）⚠️ **v3.3 新增**

### 驳回场景
```
司库财务审核不通过 → 驳回给匠心（上一个执行节点）
执矩安全审核不通过 → 驳回给匠心（上一个执行节点）
```

### 驳回操作（OCC 面板）
```
1. 访问 OCC 任务页面
2. 找到待审核任务
3. 点击 [驳回] 按钮
4. 填写驳回原因（必填）
5. 点击 [提交]
```

### 驳回操作（API）
```bash
curl -X POST http://localhost:3000/api/bce/tasks/{任务 ID}/reject \
  -H "Content-Type: application/json" \
  -d '{
    "operator": "司库",
    "reason": "预算超支，需要重新评估技术方案"
  }'
```

### 驳回后流程
```
1. 系统根据流转规则判断回退节点
2. 任务状态 → 🟢 进行中
3. 负责人收到驳回通知
4. 负责人重新执行任务
5. 完成后再次提交审核
```

### 流转规则统一适用
```
⚠️ 重要：驳回回退遵循统一的流转规则

不是硬编码回退，而是通过规则引擎判断下一节点
所有任务类型（技术/财务/安全/验收）都适用同一套规则
```

### 驳回通知
- ✅ **通知被驳回人**："司库驳回了你的任务，原因：预算超支"
- ✅ **通知任务创建人**："任务被司库驳回，已回退到匠心重新执行"
- 🚨 **升级通知**：同一任务被驳回 **5 次以上** → 通知天枢 + 磊哥

### 驳回示例
```
场景：技术方案财务审核不通过

1. 匠心执行完成 → 点击 [完成]
2. 规则引擎自动流转 → 司库（财务审核）
3. 司库审核不通过 → 点击 [驳回]
4. 填写原因："预算超支 20%，需要优化方案"
5. 系统自动回退 → 匠心（上一个执行节点）
6. 匠心收到通知 → "司库驳回了任务"
7. 匠心重新执行 → 优化方案后再次提交
8. 流转历史 → 记录驳回次数（用于升级通知）
```

### 驳回次数统计
```
驳回次数 | 处理
--------|------
1-4 次   | 正常流程，通知被驳回人
≥5 次    | 🚨 升级通知天枢 + 磊哥
```

## 6️⃣ 任务验收（天枢/创建人）

### 验收操作
```
1. 访问 OCC 任务页面
2. 找到"待验收"任务
3. 查看详情和完成说明
4. 点击 [验收通过] 或 [驳回]
```

### 验收结果
- **通过**：任务状态 → ✅ 已完成
- **驳回**：任务状态 → 🟡 待开始（返回负责人）

```
负责人确认 → 系统通知创建人："匠心已确认任务"
```

### 完成反馈
```
负责人完成 → 系统通知创建人："匠心已完成任务，请验收"

### 验收反馈
```
天枢验收 → 系统通知负责人："天枢已验收通过"

| 状态 | Emoji | 含义 | 操作 |
|------|-------|------|------|
| **待开始** | 🟡 | 任务已分配，等待确认 | 确认接收 |
| **进行中** | 🟢 | 已开始执行 | 执行任务 |

## ❓ 常见问题

### Q1: 收不到通知怎么办？
```
1. 检查飞书通知设置
2. 检查 sessions_send 连接
3. 访问 OCC 协作页面查看待确认任务
4. 联系匠心排查
```

### Q2: 如何确认任务？
```
推荐方式：OCC 协作页面点击 [确认接收] 按钮
快捷方式：飞书群回复"收到 + 姓名"
```

### Q3: 任务无法完成怎么办？
```
1. 标记为"阻塞"状态
2. 填写阻塞原因
3. @创建人或天枢协助
```

### Q4: 自动流转如何触发？
```
完成任务时自动触发规则引擎
根据任务类型自动转交下一节点
```
